# FOC 机械臂热设计工作台设计

## 目标

把 ThermalForge 已有的 FOC 机械臂肘关节工程输入、筛选级热评估、GPT 工程决策、Rodin 整体模型与 Bang 分件模型，统一成一个可运行、可审计的本地工程工作台。用户进入页面后应能直接看到真实 GLB、完整脱敏后端输出、设计依据、风险和下一步验证任务，而不是静态 SVG 原型。

## 方案选择

采用“可复现实验台”方案：保留已有 JSON/GLB 作为可重复演示快照，同时提供通用模型代理和刷新工程分析的入口。它位于“只嵌入一个 GLB 查看器”和“立刻构建完整异步任务平台”之间：前者缺少工程链路，后者会引入数据库、队列和复杂状态管理，超出当前演示需要。

本轮把六轴机械臂中的高热负载肘关节模组作为热设计对象。整机 FOC 架构通过关节编号、PMSM、逆变器、编码器和谐波减速器关系表达；热结果聚焦持续 120 W、峰值 220 W 的关节模组。页面必须明确现有热评估是集总参数筛选，不是 CFD；Rodin/Bang 输出是概念网格，不是可制造 CAD。

## 架构与数据流

1. `.env` 在服务端保存 OpenAI-compatible 网关和 Hyper3D 凭证，API 只公开模型名及配置状态。
2. `FocDemoRepository` 读取 `outputs/foc_robot_arm_backend_output.json` 与本地资产目录，对上游下载 URL、订阅键和可能的签名查询参数做递归脱敏。
3. `GET /api/v1/foc-demo` 返回面向界面的快照：输入、Brief、热指标、设计理由、阶段事件、资产清单和能力边界。
4. `GET /api/v1/foc-demo/raw` 返回完整但脱敏的原始后端 JSON，供控制台审计。
5. `GET /api/v1/foc-demo/assets/{name}` 只允许白名单目录中的 GLB/WEBP/JPG，防止路径穿越。
6. `POST /api/v1/foc-demo/reasoning` 通过配置的 `gpt-5.6-sol` 生成结构化工程决策，并把最终 JSON 保存为本地演示产物；不展示隐藏思维链，只展示可审计的依据、权衡、风险和验证任务。
7. 前端使用 React Three Fiber 加载 Rodin/Bang GLB，遍历单个 Bang GLB 内的 mesh，支持旋转、缩放、整体/分件切换和热路径标注。

## 界面方向

视觉方向为“航空试验台 / 热工程任务控制台”：深石墨底色、氧化铜橙表示热端、冷却液青表示冷端，辅以细密坐标网格、窄体标题和等宽数据标签。页面主视觉是大尺寸真实 WebGL 模型，不使用紫色渐变、卡片堆叠或伪 CFD 云图。

布局包含四个区域：顶部显示模型连接与当前运行状态；左侧是 FOC 输入和 7 阶段事件；中间是 3D 模型及整体/分件、自动旋转、线框与复位控制；右侧是“为什么这样构建”的工程决策；底部抽屉展示人类可读事件或完整脱敏 JSON。小屏幕按输入、3D、理由、输出顺序纵向排布。

## 错误、安全与边界

- 密钥仅在服务端 `.env`，不进入前端 bundle、响应、日志或文档。
- 完整输出默认递归删除 `authorization`、`api_key`、`subscription_key` 等敏感字段，并清理远程 URL 查询参数。
- 本地资产路由使用固定根目录和文件名白名单，拒绝 `..`、绝对路径与未知扩展名。
- 上游模型失败时保留当前本地快照，界面明确显示错误而不是用假成功替代。
- 生成理由必须区分输入事实、工程假设和待验证事项；不得声称完成 CFD/FEA 或制造验证。

## 测试

后端以测试先行覆盖：快照契约、脱敏、资产白名单、模型配置、Hyper3D 余额和理由解析。前端以 Vitest/Testing Library 覆盖加载、错误、模型切换、理由/原始输出标签和敏感字段不渲染。最后执行隔离插件的 36+ 项 pytest、前端测试、类型检查、生产构建，并在本地浏览器验证真实 GLB 已加载且主要交互可用。
