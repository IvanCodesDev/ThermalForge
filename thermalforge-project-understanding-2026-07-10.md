# ThermalForge 项目统一理解与方向校准

> 日期：2026-07-10  
> 范围：综合 `ziye-context-os/04-Projects/lunar-hackathon-2026/` 全部关键项目档案、原始 `thermal_forge_mvp.py` 及其历史输出，并与当前 `Desktop/thermalforge` 工程现状对照。  
> 本文只做理解与方向校准，不代表已经开始下一轮工程改造。

---

## 1. 一句话结论

ThermalForge 当前最准确的定位，不是“任意三维结构自动生成器”，也不是“实时 CFD 或通用 CAE 替代品”，而是：

> **面向高功率、空间受限、异形硬件的 AI 热设计 Copilot：把器件工况与制造约束转成受控的参数化散热候选，使用快速工程模型完成交互式反馈、自动搜索与排序，再把少量优选方案交给 CAD、CFD、FEA 和实测流程验证。**

Agent 是入口和编排层；参数中枢、参数化几何和确定性评估器才是产品内核。

---

## 2. 我对产品本质的理解

ThermalForge 真正要解决的不是“画一个漂亮散热器”，而是缩短以下工程链路：

```text
模糊需求 / 器件工况
→ 结构化约束
→ 判断是否需要结构改造
→ 选择结构族与参数空间
→ 批量生成候选
→ 快速热评估与制造风险筛选
→ 多目标排序
→ 导出候选、假设和验证任务
→ CFD / FEA / 实测校准
```

它最有价值的三个能力是：

1. **参数中枢**：把自然语言、尺寸、功耗、热限制、材料、制造方式和冷却介质转为统一契约；
2. **受控设计空间**：叶脉、平行/蛇形/歧管流道、pin-fin、导热桥等结构族都能由有限参数确定；
3. **分层评估**：先用毫秒到秒级快速模型缩小设计空间，再异步调用高保真仿真。

因此产品的核心资产不是某一个叶脉外形，而是：

- 参数与约束体系；
- 模板化结构生成器；
- 经过校准的评估模型；
- 优化运行记录与真实案例；
- 与 CAD/CAE/实测流程的集成。

## 3. 产品定位的历史演化

Context OS 文档记录了四次明显演化：

### 3.1 黑客松早期：具身智能热设计 Agent

目标是输入机器人关节/电机参数，自动产生 20–50 个候选，评估温度、重量与制造性，展示 3D 并导出 STL/STEP。

### 3.2 中期：国产具身结构级热设计平台

通过“具身场景 + 非专家 UX + 国产合规”避开通用 CAE 求解器正面竞争。

### 3.3 宇树反馈后：参数化热设计工具 / CAE 插件

方向发生重要收敛：

- 不卖 3D 打印零件；
- 不替客户承担承力、安全和最终认证责任；
- 算法降损优先，结构改造用于补尾；
- 输出设计初案、参数、排序和验证建议；
- 高保真结论交给客户 CFD、FEA 和实测。

### 3.4 当前可统一为：通用异形热设计 Copilot

能力横向面向“高功率、异形、空间和重量受限”的硬件；具身执行器、无人机动力模块可以作为初始垂直场景，而不是永久边界。

这四个阶段不是互相否定，而应被整理成分层架构：

- **交互入口**：AI Agent；
- **核心能力**：参数化热设计；
- **部署形态**：Web App、CAD/CAE 插件、API、私有化；
- **初始场景**：具身执行器与无人机动力；
- **验证层**：外部 CFD/FEA/实测。

- 材料体积；
- 扩散因子；
- 面积增强系数。

### 4.3 热评估

已有 `SimulationBackend` 抽象和可用的集总热阻后端，可以做快速相对估算；真实外部仿真后端仍是占位。

所以当前能力是：

当前 `optimize_leaf_direction()` 已经能：

```text

→ 排序 Top-K
```

这是一条真实存在的自动候选搜索骨架，但目前：

- 主要针对叶脉方向与分叉角；
- 使用枚举式候选，不是完整 Optuna study；
- 依赖快速近似热模型；
- 审美是启发式规则；
- 流道尚未形成同等成熟的自动优化入口；
- 没有跨轮次持久化 study 和实时连续调优。

### 4.5 API 与外部模型

后端已具有 FastAPI 路由，并统一接入：

- GPT-5.5 Responses；
- GPT Image 2；
- Hyper3D / Rodin；
- 根目录 `.env` 配置管理；
- 离线 Mock 测试。

这些外部模型适合承担需求理解、概念图、概念外形等角色，但不能替代精确 CAD 和物理求解器。

## 5. 原始 `thermal_forge_mvp.py` 到底做到了什么

原始脚本在形式上实现了一条完整闭环：

```text
固定示例输入
→ 五类结构参数采样
→ CadQuery 或 mock 几何
→ 集总热阻估算
→ Optuna 三目标搜索
→ 综合排序
→ JSON / Markdown 输出
```

它包含：

- 叶脉；
- 平行结构；
- 人字结构；
- 波浪/S 型结构；
- 水滴阵列；
- Optuna TPE；
- 温度、质量和可制造性三个目标；
- 每构型结果和全局 Top-5 报告。

因此，“自动改变参数 → 评估 → 排序 → 记录”的**流程原型确实存在**。

但是，它不是一个有效的工程优化闭环，存在几个阻断性问题。

### 5.1 CadQuery 基本没有真正运行

历史输出中的所有候选都是：

```json
"cad_available": false
```

脚本使用 `import cq`，与标准 CadQuery 2.x 的 `import cadquery as cq` 不一致。即使修正导入，多处工作平面 offset 和空间定位方法也存在 API 风险。

### 5.2 没有导出 STL 或 STEP

源码没有实际 exporter 调用。文档所称“输出 STL 模型”与代码不一致。

### 5.3 所谓流道多数是实体翅片或柱阵列

平行、人字、波浪和水滴生成器主要执行实体 `union`，没有通过 `cut` 形成内部空腔，没有入口/出口，也没有流体域。

因此它更接近不同形式的散热翅片，不是完整 PCHE/液冷流道模型。

### 5.4 真实 CAD 没有进入评估器

即使 CadQuery 成功，优化器也只保留：

```json
{"cad_available": true}
```

没有从实体提取真实体积、表面积、质量、水力直径、湿周或流道长度。结果仍依赖通用近似公式。

### 5.5 质量换算错误 1000 倍

历史输出中，约 80 mm 级铝合金器件被估算为数百千克。例如：

```text
体积：254781.5 mm³
报告质量：687910.1 g
正确数量级：约 687.9 g
```

错误来源是将 `mm³ × kg/m³` 除以 1000，而正确换算应乘 `10^-6`。

### 5.6 最终排序奖励“较差可制造性”

Optuna 第三目标是 `100 - manufacturability_score`，最小化逻辑正确；但最终综合分又把该逆向值正向加分，导致可制造性越差，综合分反而越高。

### 5.7 历史结果已经退化

2026-07-08 的输出表明：

- 运行耗时 7.5 秒；
- 报告称 100 个候选；
- 所有候选 `cad_available=false`；
- 所有候选超温；
- 所有可制造性总分为 0；
- 所有综合分都为 30；
- 全局 Top-5 只是前五个 `vein` trial，并不是有效优选结果。

所以历史脚本**确实运行过**，但运行成功不等于优化有效。它应被当作流程参考和失败回归样本，而不是当前能力证明。

## 6. “叶脉和网道能否自动调优”的答案

### 6.1 叶脉：可以做，而且已有局部基础

叶脉已具备较明确的参数空间：

- 结构原型；
- 主干数量；
- 分支层级；
- 分叉角；
- 分支比例；
- 主干与末端宽度；
- 通道深度；
- 流向；
- 冷却介质。

当前工程已能对部分参数做自动候选搜索。下一阶段可以把整个参数空间接入持久化 Optuna，并确保每个参数真正影响几何和评估结果。

### 6.2 “网道”需要先精确定义

目前“网道”可能对应至少三类不同对象：

1. **流道**：平行、蛇形、歧管、pin-fin，用于气流或液流；
2. **晶格/TPMS 网络**：Gyroid、Diamond、Voronoi 等体积换热结构；
3. **网络状导热筋**：没有内部流体，仅增强扩散和表面积。

这三类结构的参数、几何生成方式和物理目标完全不同，不能共用一个模糊结构类型。

### 6.3 自动调优必须满足四个条件

要把“参数会变化”升级为“参数得到有效优化”，必须保证：

1. 参数确实改变几何；
2. 几何变化被测量并进入评估；
3. 评估目标与冷却介质一致；
4. 全部 trial、假设、版本和结果可以复现。

当前原始 MVP 在第 2、3、4 点上都不充分。

### 6.4 气冷和液冷必须分开评价

气冷至少需要：

- 风速或风扇曲线；
- 翅片间距与阻塞；
- 流向；
- 风阻；
- 对流换热系数相关式。

液冷至少需要：

- 流量或入口压力；
- 入口温度；
- 冷却液物性；
- 水力直径；
- 雷诺数、努塞尔数；
- 压降；
- 泵功；
- 流量均匀性；
- 密封与最小壁厚。

只优化“最高温度 + 质量 + 表面积”不足以完成液冷流道优化。

## 7. “实时优化”应该如何定义

建议将“实时”拆成三个等级，不要混用：

### 7.1 即时估算（Instant Estimate）

用户拖动参数后，快速热模型和几何统计在毫秒到秒级更新。

这是当前最容易实现、也最适合产品交互的实时能力。

### 7.2 交互式设计搜索（Interactive Design Search）

后台 Optuna 持续提出候选，前端通过 SSE 展示：

- 当前轮数；
- 最佳温度；
- Pareto 前沿；
- 候选几何；
- 约束淘汰原因。

响应时间是秒到分钟级。这可以称为“连续优化”或“近实时搜索”。

### 7.3 高保真验证（High-Fidelity Validation）

Top-N 候选异步进入 CFD/FEA，耗时分钟到小时。结果回填优化数据库，用于：

- 校正快速模型；
- 训练代理模型；
- 调整下一轮搜索；
- 建立真实案例库。

它不能称为实时 CFD。

因此推荐的双层或三层架构是：

```text
快速解析模型：实时交互与大规模筛选
代理模型：秒级高质量复评（有足够数据后）
CFD / FEA：异步高保真验证
```

## 8. “CCF”最可能是什么

结合上下文，用户口述的“CCF”最可能是 **CFD**：

> Computational Fluid Dynamics，计算流体力学。

CFD 用于求解流体速度、压力、温度和换热，适合验证气冷与液冷结构。

相关概念边界：

- **CFD**：流动、压降、对流换热；
- **FEA**：结构强度、变形、应力、热应力；
- **CAE**：计算机辅助工程总称，包含 CFD、FEA 等；
- **拓扑优化**：在约束下寻找材料/流道布局，不等同于 CFD；
- **CCF**：在当前 ThermalForge 语境下没有明显对应的核心术语。

除非老板指的是其他团队内部缩写，否则后续应按 CFD 理解。

## 9. 任意 3D 基壳与多层热功能结构的判断

这个方向在工程上成立，也比单一平板叶脉/流道更有长期价值，但它不是当前 MVP 的自然小改动，而是下一代几何内核。

### 9.1 建议的层栈模型

```text
L0 原始承力/设备壳体
L1 界面与导热扩散层
L2 换热或流动结构层
L3 密封/封盖层
L4 外观皮肤与制造修整层
```

不同冷却形式占用不同层：

- 自然/强制风冷：L2 主要是外部翅片、晶格、扰流结构；
- 液冷：L2 是内部流道，L3 必须承担封盖与密封；
- 热管/均热板：L1/L2 需要独立的等效模型；
- 多材料导热桥：主要发生在 L1。

### 9.2 需要的新几何能力

- STEP/STL/OBJ 输入；
- B-rep 或 mesh 几何解析；
- 法向偏置与厚壳；
- 曲面分区；
- UV/参数域映射或测地线生长；
- 特征贴附；
- 布尔加、布尔减；
- 流道贯通性检查；
- 封盖和水密性检查；
- 自交、最小壁厚、碰撞和装配避让；
- 面积、体积、质量和水力参数提取。

### 9.3 正确的阶段划分

- **当前 MVP**：平面、圆柱面或少数标准包络；
- **近期**：用户提供封闭壳体，在指定面区贴附参数化特征；
- **中期**：多层规则壳栈、液冷封盖、TPMS/晶格区域；
- **远期**：任意复杂曲面、拓扑优化、多材料联合优化。

Hyper3D 可用于概念外观和造型探索，但精确水密结构必须由 CadQuery/OpenCascade/FreeCAD 等确定性 CAD 内核产生。

## 10. 审美应该放在哪里

审美可以成为 ThermalForge 的差异点，但不能与基础物理约束混成一个黑箱分数。

建议三级审美机制：

1. **几何规则**：对称性、连续性、方向一致性、密度梯度、节奏和边界贴合；
2. **工业设计约束**：品牌语言、视觉重心、接口隐藏、表面完整性；
3. **视觉复评**：只对已经通过热、制造和结构闸门的 Top-N 候选进行 GPT Image 2/视觉模型评审。

排序顺序必须是：

```text
硬约束淘汰
→ 热/流动/重量 Pareto
→ 制造与安全闸门
→ 审美复评
```

不能让漂亮方案越过密封、壁厚、压降或温度约束。

## 12. 我认为下一阶段最合理的顺序

在方向确认后，工程执行应按以下顺序，而不是直接冲向任意 3D 壳体：

### 阶段 A：修复并复现可靠的自动优化闭环

- 不照搬原始 `thermal_forge_mvp.py`；
- 复用它的流程概念，修正单位、评分和目标；
- 为叶脉与流道建立独立搜索空间；
- 确保参数真正影响几何和评估；
- 持久化全部 trial；
- 输出真实 Pareto front；
- 建立基准测试和回归样例。

### 阶段 B：把流道升级为真正的流热结构

- 明确气冷/液冷模型；
- 加入压降、泵功、流量均匀性；
- 生成入口、出口和贯通流道；
- 加入最小壁厚、封盖和泄漏风险约束。

### 阶段 C：接入精确 3D CAD

- 先支持平面、圆柱和少量标准壳体；
- 提取实体面积、体积、质量和流道水力参数；
- 真正导出 STEP/STL；
- 保持几何后端可替换。

### 阶段 D：实现交互式连续优化

- Optuna 使用 SQLite/RDB 持久化；
- 后端任务状态机；
- SSE 推送 trial 和 Pareto 更新；
- 支持暂停、恢复、追加 trial；
- Top-N 异步 CFD 复核。

### 阶段 E：任意基壳和多层功能壳

- 壳体导入、面区选择、偏置、特征贴附；
- 多层壳栈；
- 晶格/TPMS；
- 水密和制造性；
- CFD/FEA 联合闸门。

## 13. 需要老板确认的方向性问题

以下问题会直接决定下一版目录和数据契约，建议在开始工程改造前确认：

1. **主叙事**：优先“具身执行器热设计”，还是“通用高功率异形热设计 Copilot”？
2. **短期目标**：先把二维/标准包络参数优化做可靠，还是立即进入三维 CAD？
3. **网道定义**：流道、晶格/TPMS，还是网络状导热筋？
4. **冷却优先级**：先做好气冷，还是液冷必须同步进入第一版？
5. **实时定义**：参数滑块即时重算、后台 Optuna 连续搜索，还是还要求 CFD 在线回填？
6. **双层架构**：是否接受“快速模型实时筛选 + CFD 异步复核”作为正式产品架构？
7. **输出责任边界**：产品输出概念候选、原型可制造结构，还是要承担工程可采用结论？
8. **审美权重**：审美只对 Top-N 复评，还是要进入每个 trial 的实时目标？

---

## 14. 最终判断

ThermalForge 的方向没有偏，但需要把“长期愿景”和“当前闭环”清楚分开：

- 长期愿景可以是任意三维基壳上的多层热功能结构 Agent；
- 当前内核应先成为可信、可复现的参数化热设计与优化系统；
- 原始 MVP 证明了自动化流程的形态，但没有证明物理、CAD 和优化结果有效；
- 当前桌面工程在参数中枢、快速评估、匹配和接口架构上比原始 MVP 更扎实；
- 下一步最重要的不是堆更多结构，而是让每个参数、几何、指标、约束和结果真正闭环；
- CFD 应是高保真验证后端，而不是同步交互内核；
- “实时优化”应定义为快速模型驱动的交互式搜索和持续 Pareto 更新。

推荐对外使用的简洁表述：

> **一句话定义工况，几分钟筛出值得验证的散热方案。**

英文：

> **From thermal requirements to validated design candidates.**
